# 文件管理排序功能设计

## 背景

文件管理页面当前由服务端按“目录置顶、名称升序”排序后分页，客户端直接展示当前页数据。需要增加按名称、大小、修改时间选择排序的能力，同时保留目录置顶规则，并确保打开文本文件时仍通过原文件路径加载内容。

## 目标

- 在文件管理页面提供“名称、大小、修改时间”三个排序选项。
- 默认使用名称排序，行为与现有列表一致。
- 大小排序时，目录始终置顶，文件按大小从大到小排列。
- 修改时间排序时，目录始终置顶，按修改时间从新到旧排列。
- 同一排序字段值相同时，使用名称升序作为稳定的兜底顺序。
- 排序发生在服务端分页之前，保证多页目录的全局排序正确。
- 保持文件的原始 `path`，文本编辑器仍调用原有读取接口加载内容。

## 非目标

- 不改变文件实际存储位置、路径校验、读取和保存接口。
- 不新增排序方向切换；大小和修改时间固定为降序，名称固定为现有升序。
- 不持久化排序选择到本地存储；页面重新加载后恢复默认名称排序。
- 不改变现有搜索接口的搜索范围、深度和返回上限。

## 方案

### 服务端排序与分页

文件列表接口 `/api/files/list` 新增可选查询参数 `sortBy`，取值为 `name`、`size`、`modified`，缺省值为 `name`。排序比较器位于服务端 `server/src/utils/fileSorting.ts`，以符合现有服务端 `rootDir` 和 Docker 输出结构；列表和搜索接口统一复用该比较器。服务端读取目录项并取得 `name`、`type`、`size`、`modified` 后，先执行排序，再按照现有 `page` 和 `pageSize` 分页。

排序比较规则：

1. 类型比较优先，目录排在文件之前。
2. `name`：按现有 `localeCompare(name, 'zh-CN')` 升序。
3. `size`：数值降序；相同大小按名称升序。
4. `modified`：时间戳降序；相同修改时间按名称升序。

收到未知 `sortBy` 时回退为名称排序，保持接口兼容和现有默认行为。

### 客户端交互

文件列表表头保留名称、大小、修改时间列，在表头区域增加与现有面板风格一致的排序选择控件。默认值为 `name`，选择变化时从第 1 页重新请求当前目录，并传递当前 `sortBy`。当前页切换、刷新、上传和文件操作完成后的重新加载都沿用当前排序选择。

搜索结果继续由现有搜索接口返回，接口接收当前排序选择并在返回上限截取前完成排序。搜索结果不参与目录分页；搜索时仍隐藏批量选择控件，且不改变原有文件路径显示和点击行为。

排序只影响列表顺序，不改变每一项的 `path`。因此点击文件进入编辑器时，仍调用 `readFile(filePath)`，编辑器加载原本路径上的文本内容。

## 数据流

1. 页面初始化排序状态为 `name`，从查询参数或本地存储恢复当前目录路径。
2. 页面调用 `fileStore.fetchFiles(path, 1, sortBy)`。
3. store 将 `sortBy` 传给 `apiClient.listFiles`。
4. 服务端完成目录置顶、字段排序和分页后返回文件项。
5. 页面根据服务端返回顺序展示列表；编辑文件仍使用返回项的原始 `path`。
6. 选择新的排序字段后，清空当前选择项并重新请求第 1 页。

## 错误处理与兼容性

- `sortBy` 不传或值无效时，服务端按名称排序。
- 既有客户端调用 `listFiles(path, page, pageSize)` 不应因新增参数而失效；新增参数置于可选参数位置或通过兼容的参数方式传递。
- 文件属性读取失败的项目继续按现有逻辑跳过。
- 网络错误继续使用现有通知组件提示，不使用浏览器消息或对话框。

## 验证

- 使用排序比较逻辑的测试验证三种排序顺序、目录置顶和相同值名称兜底。
- 客户端和服务端 TypeScript 构建通过。
- 手动确认切换排序后分页从第一页开始，翻页顺序保持一致。
- 手动确认打开文本文件仍能加载原路径内容，排序不影响编辑和保存。
- 测试代码在验证完成后删除，遵循项目约定。
